## Repo Shape

This is a PySide6 desktop application with strict inward dependencies:

- `domain` -> typed state and ECU models
- `parsers` -> INI / MSQ / project ingestion
- `transports` -> serial / TCP / UDP / mock I/O
- `comms` -> Speeduino raw protocol, XCP, framing, CRC, controller abstractions
- `services` -> business logic and orchestration
- `plugins` -> extension API scaffolding
- `ui` -> thin PySide6 widgets

Rules:

- UI should not talk directly to transports.
- Protocol code should not depend on Qt.
- Parsers should return typed models, not UI-shaped blobs.
- INI constructs should be compiled once into stable layout/page models before the UI consumes them.

## Post-Roadmap Direction

Treat the current Python application as the reference product while workflow, layout, parser, and compatibility behavior are still being proven against real artifacts.

Longer-term direction after the main roadmap/backlog:

- stabilize the product model and fixture coverage first
- define an owned tune/definition/project contract rather than remaining permanently coupled to legacy external formats
- evaluate a native shared core, with C++ currently the leading direction, for parsers, protocol, validation, and packaging-sensitive logic
- keep any future autotune engine deterministic and operator-reviewable first; treat ML as an optional assistive layer for sample classification, confidence scoring, anomaly detection, or coverage analysis rather than as the first correction engine

Native-format findings to preserve in future planning:

- release compatibility is still tightly coupled to signature families, page sizes, offsets, and parser behavior
- constructs such as `lastOffset` and indirect table identity through layout/menu naming are legacy compatibility burdens
- decompiled TunerStudio sources show application-managed INI sections, reinforcing the need to separate firmware storage data from app-owned metadata
- preferred long-term direction is semantic native files plus a separate firmware export layer rather than a one-for-one recreation of legacy INI/MSQ behavior
- if/when native files are introduced, prefer `JSON5` for human-authored definitions and `JSON` for tune/project data unless stronger constraints emerge

Do not let speculative native-port work distort near-term Python architecture. If code is likely to survive a future C++ migration, keep boundaries clean between schema/model logic, protocol logic, services, and UI.

## INI Parser Notes

The INI parser handles:
- `lastOffset` for sequential array/scalar offset tracking — confirmed working against release artifacts
- `#if`/`#else`/`#endif`/`#set`/`#unset` conditional compilation — implemented and tested
- `active_settings: frozenset[str]` parameter on `IniParser.parse()` for project-level flags (e.g. `LAMBDA`, `mcu_teensy`)
- File-scope `#set`/`#unset` set the defaults; caller-supplied `active_settings` always overrides them
- `blockingFactor` and `tableBlockingFactor` parsed and consumed by write chunking
- `[SettingGroups]` parsed into `EcuDefinition.setting_groups: list[SettingGroupDefinition]` (boolean flags and multi-option groups)
- `active_settings` persisted in project file as `activeSettings=LAMBDA,mcu_teensy`; loaded by `ProjectParser`; passed through `DefinitionService.open_definition()`
- "Definition Settings" toolbar button opens a dialog showing all setting groups; updates project and reloads definition on accept
- `[CurveEditor]` parsed into `list[CurveDefinition]` on `EcuDefinition` (34 curves in production INI)
- `[GaugeConfigurations]` parsed into `list[GaugeConfiguration]` (70+ named gauges with category, channel, title, units, lo/hi display range, warn/danger thresholds, digits)
- `[FrontPage]` parsed into `front_page_gauges: list[str]` (8 gauge slots) and `front_page_indicators: list[FrontPageIndicator]` (40+ boolean/bitfield indicator expressions)
- `[LoggerDefinition]` parsed into `list[LoggerDefinition]` (4 loggers: tooth, 3× composite); `\$tsCanId` substituted with `\x00\x00`; `\xNN` escape sequences decoded to bytes; `calcField` lines skipped; `record_count` derived from `dataLength` and `record_len` with kind-aware logic (tooth = bytes ÷ record_len, composite = declared record count)
- `[ControllerCommands]` parsed into `list[ControllerCommand]` on `EcuDefinition`; `\xNN` escapes decoded; all production commands are 3-byte entries: `E` + subtype + param

## Known Fragile Areas

Track these explicitly. They are not bugs yet but carry real regression risk:

1. **Blank table when tune lacks data — fix available on both sides**: by default `MsqWriteService.save()` / C++ `write_msq_text()` still drops staged values for constants absent from the source XML. Pass `insert_missing=True` (Python) or call `write_msq_text_with_insertions()` (C++, sub-slice 89) to inject the missing `<constant>` nodes into the first `<page>` element instead. `tests/unit/test_msq_write_insert_missing.py` + `tests/unit/test_cpp_msq_write_insert_missing_parity.py` + 9 C++ doctest cases cover tables, scalars, staged-only values, preservation of existing constants, skip-already-present, compose-with-update, idempotent re-save. Default behavior is unchanged so existing fixtures stay byte-stable.
2. **Table identity indirection**: `afrTable` (constant name) → `afrTable1Tbl` (table editor ID) → `afrTable1Map` (map ID) → `zBins = afrTable` (data pointer). This 4-hop naming is implicit; a rename anywhere silently breaks the link.
3. **Endianness field consumer added but byte paths unchanged**: `EcuDefinition.is_little_endian()` and `byte_order()` are now the canonical helpers (default little, case-insensitive, unknown values fall back to little — see `tests/unit/test_endianness_consumer.py`). The actual Speeduino raw-protocol read/write code still hard-codes little-endian; the helpers are in place so future byte-order-aware code paths have one place to query.
4. **U16P2 experimental page 2**: the experimental INI itself notes "the TS page-2 transport is native U16, but the main fueling path still consumes VE through byte-returning lookups, so this is not yet a proven end-to-end high-resolution fueling mode." Our test coverage for U16 page-2 tables is limited.
5. **`arrayValue()` in visibility expressions**: now implemented and tested. `IniParser` parses `array` kind output channels and `defaultValue` lines into `EcuDefinition.output_channel_arrays`; `VisibilityExpressionService.evaluate()` accepts an optional `arrays` dict and resolves `arrayValue(array.NAME, index)` and `arrayValue(NAME, index)` calls. `TuningWorkspacePresenter._is_page_visible()` passes `self.definition.output_channel_arrays`. Unknown functions fail-safe to 0.0. Callers that lack a definition (scalar editor, page validation) still pass `None` — field-level visibility expressions don't use `arrayValue()` in practice.
6. **Real-release round-trip**: covered by `tests/unit/test_release_round_trip.py` (19 tests). Scalar edit, table cell edit, lastOffset-derived table preservation (`afrTable`), multi-edit, XML format sanity. Both fixture artifacts now checked in to `tests/fixtures/`.

## Comms Notes

Current real support:

- Speeduino legacy raw serial path: connect, page read, parameter write, burn, runtime polling
- TCP/WiFi path via `TcpTransport` with Speeduino new-protocol framing (`[u16 LE len][payload][u32 LE CRC32]`); Airbear **v0.2.0** bridge on port 2000; mDNS `speeduino.local`; EcuHub UDP auto-discovery on :21846; Airbear `DASH_ECHO` mode returns `RC_BUSY_ERR (0x85)` when dashboard holds the UART mutex — TCP transport must recognise this and back off/retry instead of treating it as framing corruption
- `SpeeduinoControllerClient._send_data_command()` / `_recv_data_response()` detect framing via `getattr` — framed over TCP, raw over serial; Q/S handshake stays raw
- `fetch_logger_data(logger_def)` — live tooth/composite log capture
- simulator JSON and mock paths
- partial XCP support

Do not describe XCP as feature-complete.
